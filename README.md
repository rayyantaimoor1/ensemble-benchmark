# Benchmarking Ensemble Learning Algorithms on High-Dimensional Spatial Data

A rigorous, empirical comparison of four ensemble-learning paradigms — **Random Forest**
(bagging), **Gradient Boosting** (vanilla boosting), **XGBoost** (optimized boosting), and
**LightGBM** (leaf-wise histogram boosting) — on the UCI **Forest Covertype** dataset
(581,012 rows, 54 features, 7-class classification).

## 1. Problem Statement

In supervised machine learning, ensemble methods represent the state of the art for tabular
data, but there is a fundamental trade-off between a model's computational complexity
(training time / memory) and its predictive accuracy. Standard bagging and boosting
algorithms often struggle to scale as datasets grow to hundreds of thousands of rows.
Modern optimized gradient-boosting frameworks like XGBoost and LightGBM claim to solve this
through engineering choices such as regularization, leaf-wise tree growth, and histogram
binning — but direct, empirical, visual benchmarks of that trade-off on a single complex
dataset are less common. This project runs that comparison end-to-end and reports it with
reproducible code and figures.

## 2. Dataset

**[Forest Covertype](https://archive.ics.uci.edu/dataset/31/covertype)** (UCI ML Repository, id=31)

| | |
|---|---|
| Task | Multi-class classification (7 forest cover types) |
| Instances | 581,012 |
| Features | 54 (10 continuous topographic/hydrological/meteorological + 44 binary one-hot wilderness/soil) |
| Challenge | High dimensionality, class imbalance, large row count |

The dataset is downloaded automatically on first run (via `ucimlrepo`, falling back to
scikit-learn's OpenML fetcher if unavailable) and cached locally under `data/`.

## 3. Research Questions

1. **Computational efficiency** — How do training time and memory scale with data volume?
2. **Predictive performance** — How do the models compare on Accuracy and Macro F1 (given class imbalance)?
3. **Overfitting & generalization** — How do train-vs-validation learning curves differ between bagging (variance reduction) and boosting (bias reduction via sequential correction)?
4. **Feature interpretation** — How does feature-importance ranking differ across paradigms?

## 5. Methodology

**Phase 1 — Data pipeline:** stratified 70/15/15 train/val/test split; `RobustScaler` fit on
training-split continuous features only (robust scaling chosen over standard scaling because
several features, e.g. distance-to-hydrology, are right-skewed with outliers); binary
one-hot columns left untouched.

**Phase 2 — Model training & tuning:** all four models trained under unified baseline
conditions first, then (optionally) tuned via `RandomizedSearchCV` with stratified k-fold CV.

**Phase 3 — Benchmarking metrics:** training time, peak memory (via `tracemalloc`), inference
latency, Accuracy, and Macro F1 — recorded both on the full training set and across a
scalability sweep at 10k / 50k / 100k / 500k rows.

## 7. Setup & Usage

```bash
# 1. Clone and install dependencies
git clone <your-repo-url>
cd ensemble-benchmark
pip install -r requirements.txt

# 2. Run the full pipeline (baseline models, default scalability sweep)
python src/main.py

# 3. Optionally include hyperparameter tuning (slower)
python src/main.py --tune --tune-iter 20

# 4. Customize the scalability sweep row sizes
python src/main.py --sweep-sizes 10000 50000 100000 500000
```

Or open the notebook for a narrated walkthrough:

```bash
jupyter notebook notebooks/ensemble_benchmark_analysis.ipynb
```

Outputs land in `results/metrics/*.csv` and `results/figures/*.png`.

### Running tests

```bash
pip install pytest
pytest tests/ -v
```
